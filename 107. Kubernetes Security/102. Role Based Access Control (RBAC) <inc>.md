# Role Based Access Control (RBAC)

## Objective:
In this lab, you will implement Role-Based Access Control (RBAC) in Kubernetes to restrict user permissions. You’ll create a Role that grants read-only access to Pods and bind it to a user to test access behavior.

## Tasks:

### 1. Create a Role

In the default namespace, define a YAML manifest for a Role named pod-reader.

This Role should allow performing the following actions on the pods resource: get, watch, list

Ensure that the Role’s apiGroups field is empty (""), since Pods belong to the core API group.

---

### 2. Create a RoleBinding

Create a RoleBinding manifest named pod-reader-binding in the same namespace.

Bind this Role to the User named default (not a ServiceAccount).

The RoleBinding should correctly reference the Role (pod-reader) and the subject type should be User.

---

### 3. Apply and Verify

Apply both YAML manifests in the cluster.

Verify that the Role and RoleBinding are created in the default namespace.

Test whether the user default can successfully perform allowed actions (get, watch, list) on Pods.

Confirm that the same user is restricted from performing unauthorized actions like creating or deleting Pods.

---

### Notes:

Ensure both YAMLs include the namespace field under metadata.

Role should use apiGroups: [""] for core resources like Pods.

RoleBinding must correctly reference the Role using roleRef.

Verification is done by simulating user permissions using the kubectl auth can-i mechanism.

---

### Expected Outcome:
Role pod-reader exists in the default namespace and grants read-only access to Pods.

RoleBinding pod-reader-binding correctly binds this Role to the user default.

The user default can get, watch, and list Pods but cannot create or delete them.
