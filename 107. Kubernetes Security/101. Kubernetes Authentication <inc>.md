# Kubernetes Authentication

## Objective
Learn how to create and authenticate a new Kubernetes user (dev-user) using client certificates, configure the Kubeconfig file for access, and verify RBAC permissions that allow limited interaction with the cluster.

### Scenario
You are part of a DevOps team responsible for managing secure Kubernetes access for different users. Your task is to create a user named dev-user, configure their authentication through client certificates, and restrict their permissions using Role-Based Access Control (RBAC).

You will then verify that the user can successfully authenticate and list cluster nodes.

## Tasks
### 1. Generate a Client Certificate for dev-user

Generate a private key for the new user.

Create a Certificate Signing Request (CSR) for dev-user, ensuring the Common Name (CN) is dev-user and the Organization (O) is dev-group.

Submit this CSR to the Kubernetes cluster by defining a CertificateSigningRequest (CSR) resource in a YAML file.

The YAML should specify:
```
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata.name: dev-user-csr
spec.signerName: kubernetes.io/kube-apiserver-client
spec.usages should include client auth
```

The encoded CSR content under spec.request

Once applied, approve the CSR to generate a valid signed certificate for the user.

Store the signed certificate as dev-user.crt in the working directory (/home/user).

---

### 2. Configure the Kubeconfig File for dev-user

Add dev-user as a new entry in the Kubeconfig with the generated client certificate and private key.

Create a new context named dev-user-context that connects to the existing cluster (usually named default) and uses dev-user for authentication.

Set dev-user-context as the current context to be used by kubectl.

This ensures all future kubectl commands are executed as dev-user.

---

### 3. Assign RBAC Permissions

Define and apply an RBAC policy to allow limited access to cluster resources.

Create two YAML files:

ClusterRole file (dev-user-role.yaml)

Should allow only the verbs get, list, and watch on the nodes resource.

ClusterRoleBinding file (dev-user-rolebinding.yaml)

Should bind the above role to the dev-user identity.

Apply both YAML files to grant the necessary permissions.

---

4. Verify Authentication and Authorization

Confirm that dev-user.crt exists and is a valid certificate.

Check that the current context is correctly set to dev-user-context.

Test whether dev-user can perform allowed operations, such as listing nodes in the cluster.

If configured correctly, kubectl get nodes should work, while restricted actions should fail.

---

### Notes

Ensure all files (.key, .csr, .crt, and YAML files) are created in the home directory (/home/user) for testing compatibility.

The context name must exactly be dev-user-context, as test scripts check for this specific value.

The certificate file must be named dev-user.crt and valid per openssl verification.

All YAML definitions must use correct API versions (rbac.authorization.k8s.io/v1, certificates.k8s.io/v1).

---

### Expected Outcome

A valid signed client certificate (dev-user.crt) is generated.

The Kubeconfig file includes a working context dev-user-context.

dev-user can successfully authenticate and list Kubernetes nodes.

All three test cases pass, confirming authentication, configuration, and permissions.
