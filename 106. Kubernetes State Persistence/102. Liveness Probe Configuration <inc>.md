# Liveness Probe Configuration

## Description
In this lab, you will configure a Kubernetes Deployment that demonstrates how a liveness probe automatically restarts a container when it stops responding to health checks.
You’ll simulate an HTTP service using the busybox image and configure the liveness probe to monitor the /healthz endpoint. If the service fails to respond within a certain time frame (more than 30 seconds), Kubernetes will detect the failure and restart the Pod automatically.

This mechanism ensures that the application remains self-healing, maintaining high availability even when individual containers become unresponsive.

## Objective
To deploy an application named api-service that uses a liveness probe to monitor its own health and recover from simulated failures automatically.

### You will:
```
Use busybox to simulate a minimal HTTP service.
Configure a liveness probe that checks an HTTP endpoint periodically.
Observe how Kubernetes automatically restarts the container when the service stops responding.
```

## Tasks

### 1. Create the Deployment Configuration
```
Create a new Deployment manifest file named api-service.yaml.
Define a Deployment called api-service under the default namespace.
Set the replica count to 1 — since this test focuses on observing restart behavior rather than scaling.
Use the busybox image to simulate an HTTP server.
Inside the container, configure a simple loop that:
Listens on port 8080.
Serves a basic “Healthy” message through an HTTP response.
After a few successful requests, stops responding to simulate a service failure.
This behavior will trigger the liveness probe failure, causing Kubernetes to restart the container automatically.
```
Purpose:
This setup allows Kubernetes to monitor the container’s ability to serve HTTP responses and demonstrate self-healing behavior when the service becomes unresponsive.

### 2. Configure the Liveness Probe

Add a livenessProbe to the container specification with the following configuration:
```
Perform an HTTP GET request to the /healthz endpoint on port 8080.
Set initialDelaySeconds to 10 seconds — allowing time for the server to start before probing begins.
Set periodSeconds to 10 seconds, meaning the probe runs every 10 seconds.
Set failureThreshold to 3, so if the service fails three consecutive checks (≈30 seconds), the container
is marked unhealthy and restarted by Kubernetes.
```
Purpose:
The liveness probe continuously monitors service responsiveness, ensuring the Pod restarts automatically if the application becomes unresponsive for an extended period.

### 3. Demonstrate Pod Recovery
```
Deploy the api-service manifest to the cluster.
Observe the Pod status to ensure it starts and runs initially.
Wait for the container to stop responding (as simulated).
Kubernetes should detect the failure through the liveness probe and restart the container.
Monitor the Pod’s restart count — it should increment after Kubernetes restarts it automatically.
```
Purpose:
This demonstrates that the liveness probe effectively detected the unresponsive container and triggered a self-healing restart.

### Notes:
```
The liveness probe is designed for detecting runtime health issues, not startup delays.
When the probe fails consecutively beyond the configured failureThreshold, Kubernetes kills and recreates the container.
The initial delay ensures the first probe occurs only after the container is expected to be running properly.
The restartCount field in the Pod status confirms whether a restart has occurred due to a probe failure.
This configuration is commonly used in production systems to automatically recover from temporary application hangs or deadlocks.
```

### Expected Outcomes
After completing the lab:
```
The Deployment api-service runs a single busybox container acting as a lightweight HTTP server.
The container initially serves healthy responses but later stops responding, simulating a failure.
The liveness probe detects unresponsiveness on /healthz for more than 30 seconds and triggers a restart.
After restart, the container resumes normal operation, demonstrating Kubernetes’ self-healing capability.
The test case validates this by checking whether the restart count for the Pod is greater than 0.
```
